## 1. What is DVC and Why Use It?

**DVC** (Data Version Control) sits on top of Git and fills the gap Git cannot handle alone:

| Problem with plain Git | DVC solution |
|---|---|
| Large data files bloat the repo | DVC stores data externally; only a tiny `.dvc` pointer file is committed |
| No way to reproduce a model run | `dvc repro` re-runs only the changed stages |
| Hard to compare experiment results | `dvc metrics diff` shows metric changes between git commits |
| Pipeline steps are not documented | `dvc.yaml` is a self-documenting, version-controlled pipeline definition |

**Core idea:** Git tracks *code + metadata*; DVC tracks *data + models + metrics*.

## 3. About the Dataset

### What kind of data is it?

This project uses a **synthetic regression dataset** generated by `sklearn.datasets.make_regression`. There is no real-world CSV file to download — the data is programmatically created each time the `generate` stage runs, making the pipeline fully self-contained and reproducible.

### Why synthetic data?

| Reason | Explanation |
|---|---|
| No external dependencies | No need to download or license a dataset |
| Fully controlled | You can tune noise, sample count, and feature count via `params.yaml` |
| Reproducible | Same `random_state` always produces identical data |
| Ideal for DVC demos | Easy to see how changing `n_samples` or `noise` triggers a fresh pipeline run |

### Dataset structure

`data/raw/dataset.csv` — **500 rows × 6 columns**:

| Column | Type | Description |
|---|---|---|
| `feature_1` | float | Input feature 1 (continuous, raw scale) |
| `feature_2` | float | Input feature 2 (continuous, raw scale) |
| `feature_3` | float | Input feature 3 (continuous, raw scale) |
| `feature_4` | float | Input feature 4 (continuous, raw scale) |
| `feature_5` | float | Input feature 5 (continuous, raw scale) |
| `target` | float | Continuous output (what the model learns to predict) |

The features are drawn from a standard normal distribution. The target is a **linear combination** of some features plus Gaussian noise, so a linear model is theoretically optimal for this data.

### Dataset statistics (from a sample run)

```

count   500.00     500.00     500.00     500.00     500.00      500.00
mean      0.01       0.05       0.09      -0.00       0.02        6.59
```

### How the data flows through the pipeline

```
data/raw/dataset.csv          (500 rows, unscaled)
        │
        ▼  StandardScaler → train_test_split (80/20)
        │
   ┌────┴────────────────────┐
   ▼                         ▼
data/processed/train.csv   data/processed/test.csv
(400 rows, scaled)         (100 rows, scaled)
```

After preprocessing, every feature column has **mean ≈ 0** and **std ≈ 1** — a requirement for Ridge regression to penalise coefficients fairly.

### Controlled via `params.yaml`

```yaml
generate:
  n_samples: 500     # total rows in dataset.csv
  noise: 0.3         # noise multiplier (higher = harder problem)
  random_state: 42   # seed for reproducibility
```

Change any of these → DVC re-generates the dataset and re-runs all downstream stages automatically.

## 4. About the Model — Ridge Regression

### What is Ridge Regression?

**Ridge regression** (also called L2 regularisation or Tikhonov regularisation) is an extension of ordinary least squares (OLS) linear regression. It adds a **penalty term** to the loss function that shrinks large coefficients toward zero:

$$
\text{Loss} = \underbrace{\sum_{i=1}^{n}(y_i - \hat{y}_i)^2}_{\text{RSS (same as OLS)}} + \underbrace{\alpha \sum_{j=1}^{p} \beta_j^2}_{\text{L2 penalty}}
$$

Where:
- $y_i$ = actual target value
- $\hat{y}_i$ = predicted value
- $\beta_j$ = model coefficients
- $\alpha$ = regularisation strength (controlled via `params.yaml`)

### OLS vs Ridge — side by side

| Property | OLS (alpha=0) | Ridge (alpha>0) |
|---|---|---|
| Loss function | RSS only | RSS + L2 penalty |
| Coefficient size | Can be large | Shrunk toward zero |
| Overfitting risk | Higher | Lower |
| Feature selection | None | None (use Lasso for that) |
| Best when | Features are independent | Features are correlated or noisy |
| Solution | $(X^TX)^{-1}X^Ty$ | $(X^TX + \alpha I)^{-1}X^Ty$ |

### The `alpha` hyperparameter

`alpha` controls how aggressively coefficients are penalised:

| `alpha` value | Effect |
|---|---|
| `0` | Identical to plain OLS — no regularisation |
| `0.01` (current) | Very light penalty — coefficients nearly the same as OLS |
| `0.1` (baseline) | Mild penalty — slight shrinkage |
| `1.0` | Moderate regularisation — noticeable shrinkage |
| `100` | Heavy regularisation — coefficients pulled strongly toward zero, model underfits |

### Trained model coefficients (alpha=0.01)

```
feature_1 :  27.14
feature_2 :  79.88    ← strongest predictor
feature_3 :  31.10
feature_4 :  71.72    ← second strongest predictor
feature_5 :  10.19    ← weakest predictor
```

The magnitude of each coefficient shows how much the model weights that feature. Because the data was generated with these known relationships, a linear model recovers them nearly perfectly (R² = 0.924).

### Evaluation metrics explained

| Metric | Formula | What it tells you |
|---|---|---|
| **MAE** | $\frac{1}{n}\sum|y_i - \hat{y}_i|$ | Average absolute prediction error (same units as target) |
| **RMSE** | $\sqrt{\frac{1}{n}\sum(y_i-\hat{y}_i)^2}$ | Penalises large errors more than MAE |
| **R²** | $1 - \frac{\text{RSS}}{\text{TSS}}$ | Proportion of variance explained (1.0 = perfect, 0 = mean baseline) |

**Results on this dataset:**
```
MAE  = 25.21   → predictions are off by ~25 units on average
RMSE = 30.93   → a few larger errors pull this above MAE
R²   = 0.924   → the model explains 92.4% of the variance in the target
```

### Why Ridge for this problem?

1. The dataset is **small** (500 rows) — regularisation helps prevent overfitting.
2. The features are **mildly correlated** — Ridge is numerically stable when the design matrix is near-singular.
3. It is a **linear** ground truth — a linear model is optimal; Ridge slightly improves generalisation over plain OLS.
4. The model is **interpretable** — each coefficient directly shows the weight of a feature.

### Model artefact

The trained model is saved as `models/model.pkl` (Python `pickle` format). DVC tracks this file by its MD5 hash in `dvc.lock`, so every git commit records the exact model that was produced.

To load and use the model in a new script:
```python
import pickle

with open("models/model.pkl", "rb") as f:
    model = pickle.load(f)

prediction = model.predict([[0.5, -1.2, 0.8, 1.1, -0.3]])
print(prediction)   # e.g. [42.7]
```

## Step 2 — Define Parameters (`params.yaml`)

```yaml
# params.yaml
generate:
  n_samples: 500
  noise: 0.3
  random_state: 42

train:
  test_size: 0.2
  random_state: 42
  alpha: 0.1           # Ridge regularisation strength
```

**Why a separate params file?**  
DVC can detect when a parameter changes and automatically invalidate (and re-run) only the affected stages.  
You change `alpha: 0.1` → only the `train` and `evaluate` stages re-run. `generate` and `preprocess` are skipped.

## Step 3 — Write Pipeline Stages (`src/`)

Each Python script handles one responsibility:

### `src/generate_data.py` — Stage: `generate`
- Reads `params.yaml` → `generate.*`
- Creates a 500-row synthetic regression dataset with 5 features using `sklearn.datasets.make_regression`
- Saves to **`data/raw/dataset.csv`**

### `src/preprocess.py` — Stage: `preprocess`
- Reads **`data/raw/dataset.csv`**
- Applies `StandardScaler` (mean=0, std=1) to all feature columns
- Splits into 80 % train / 20 % test
- Saves **`data/processed/train.csv`** and **`data/processed/test.csv`**

### `src/train.py` — Stage: `train`
- Reads **`data/processed/train.csv`**
- Trains `sklearn.linear_model.Ridge(alpha=0.1)`
- Serialises model to **`models/model.pkl`** via `pickle`

### `src/evaluate.py` — Stage: `evaluate`
- Loads **`models/model.pkl`** and **`data/processed/test.csv`**
- Computes MAE, RMSE, R²
- Writes **`metrics/metrics.json`**
- Saves a residuals scatter + histogram plot to **`metrics/residuals_plot.png`**

## Step 4 — Declare the Pipeline (`dvc.yaml`)

```yaml
stages:
  generate:
    cmd: python src/generate_data.py   # shell command to run
    deps:
      - src/generate_data.py           # code dependency
    params:
      - generate.n_samples             # parameter dependencies
      - generate.noise
      - generate.random_state
    outs:
      - data/raw/dataset.csv           # output tracked by DVC cache

  preprocess:
    cmd: python src/preprocess.py
    deps:
      - src/preprocess.py
      - data/raw/dataset.csv           # depends on previous stage output
    params:
      - train.test_size
      - train.random_state
    outs:
      - data/processed/train.csv
      - data/processed/test.csv

  train:
    cmd: python src/train.py
    deps:
      - src/train.py
      - data/processed/train.csv
    params:
      - train.alpha
      - train.random_state
    outs:
      - models/model.pkl

  evaluate:
    cmd: python src/evaluate.py
    deps:
      - src/evaluate.py
      - models/model.pkl
      - data/processed/test.csv
    metrics:
      - metrics/metrics.json:
          cache: false                 # small file: keep in git
    plots:
      - metrics/residuals_plot.png:
          cache: false
```

**Key `dvc.yaml` keywords:**

| Keyword | Meaning |
|---|---|
| `cmd` | Shell command DVC will execute for this stage |
| `deps` | Files/dirs that, if changed, trigger a re-run |
| `params` | Keys in `params.yaml` that trigger a re-run when changed |
| `outs` | Files DVC caches and tracks (excluded from git) |
| `metrics` | Output files DVC compares across experiments |
| `plots` | Output files DVC can render as plots |

## Step 5 — Run the Pipeline (`dvc repro`)

```bash
dvc repro
```

`dvc repro` reads `dvc.yaml`, builds a DAG (Directed Acyclic Graph) of stages, and runs only the stages whose dependencies have changed.

**First run — all four stages execute:**
```
Running stage 'generate':
> python src/generate_data.py
[generate_data] Saved 500 rows → data/raw/dataset.csv

Running stage 'preprocess':
> python src/preprocess.py
[preprocess] Train: 400 rows | Test: 100 rows

Running stage 'train':
> python src/train.py
[train] Training Ridge(alpha=0.1) on 400 samples …
[train] Model saved → models/model.pkl

Running stage 'evaluate':
> python src/evaluate.py
[evaluate] MAE=25.2107  RMSE=30.9322  R²=0.9240
```

**Second run (no changes) — nothing runs:**
```bash

# Stage 'generate' didn't change, skipping
# Stage 'preprocess' didn't change, skipping
# Stage 'train' didn't change, skipping
# Stage 'evaluate' didn't change, skipping
# Data and pipelines are up to date.
```

## How DVC Tracks Datasets — Deep Dive

This is the most important concept. Here is exactly what DVC does:

### 1. Content Hashing
When a stage produces an output (e.g., `data/raw/dataset.csv`), DVC computes a **MD5 hash** of its contents:
```
e.g.  a3f2c1d8b4e9...  (32-character hex string)
```

### 2. The DVC Cache
DVC copies the file into its **local cache** at `.dvc/cache/`:
```
.dvc/cache/
└── files/
    └── md5/
        └── a3/
            └── f2c1d8b4e9...   ← the actual file content stored by hash
```
The cache uses the first 2 characters as a subdirectory to avoid millions of files in one folder.

### 3. `dvc.lock` — The Pipeline Snapshot
After each `dvc repro`, DVC writes `dvc.lock`. This file records **the exact hash of every dependency and output**:

```yaml
# dvc.lock (auto-generated, committed to git)
stages:
  generate:
    cmd: python src/generate_data.py
    deps:
    - path: src/generate_data.py
      md5: 9f3a...                 # hash of the script
    params:
      params.yaml:
        generate.n_samples: 500
        generate.noise: 0.3
    outs:
    - path: data/raw/dataset.csv
      md5: a3f2c1d8b4e9...        # hash of the CSV ← THIS is dataset tracking
      size: 38742
      nfiles: null
```

**Git commits `dvc.lock`** → every git commit captures the exact state of every data file.

### 4. Smart Cache Restore
When you `git checkout` an older commit, `dvc.lock` changes to the old hashes. Running `dvc checkout` then restores the *exact* old data files from cache — without re-running any code.

```bash
git checkout HEAD~1          # go back one commit
dvc checkout                 # restore old data/models from cache
```

### 5. What Git Stores vs What DVC Stores

```
Git tracks (small, text):          DVC tracks (large, binary):
─────────────────────────          ────────────────────────────
dvc.yaml          ← pipeline       data/raw/dataset.csv
dvc.lock          ← hashes         data/processed/train.csv
params.yaml       ← params         data/processed/test.csv
src/*.py          ← code           models/model.pkl
metrics/metrics.json               metrics/residuals_plot.png
```

## Step 7 — Experiment: Change a Parameter

Edit `params.yaml` — lower `alpha` to reduce regularisation:

```yaml
train:
  alpha: 0.01     # was 0.1
```

Run only the affected stages:
```bash
dvc repro
```
DVC detects that `alpha` changed → skips `generate` and `preprocess` → re-runs `train` and `evaluate` only.

Commit the new experiment:
```bash
git add dvc.lock params.yaml metrics/metrics.json
git commit -m "Experiment: Ridge alpha=0.01"
```

## Pipeline DAG

```
generate  →  preprocess  →  train  →  evaluate
(CSV)         (CSV×2)       (.pkl)    (metrics.json)
```

Each arrow represents a **data dependency**. DVC tracks whether the file at each arrow has changed (via MD5 hash) to decide what to re-run.

## Key DVC Commands Reference

| Command | What it does |
|---|---|
| `dvc init` | Initialise DVC in a git repo |

| `dvc status` | Show which stages are out of date |
| `dvc dag` | Print the pipeline DAG |

| `dvc metrics diff [ref]` | Compare metrics vs a git ref |

| `dvc pull` | Download cached data from remote storage |
| `dvc gc -w` | Garbage-collect cache entries not in current workspace |

## Summary: How Everything Connects

```
params.yaml  ──►  src/*.py  ──►  dvc.yaml  ──►  dvc repro
    │                                               │
    └── parameter hashes ──►  dvc.lock  ◄── output hashes
                                  │
                            git commit
                                  │
                    reproducible experiment snapshot
```

- **`params.yaml`** stores what you want to experiment with.
- **`dvc.yaml`** stores how stages connect.
- **`dvc.lock`** stores the exact state of all files at the time of a run.
- **Git** stores `dvc.lock`, so every commit = a fully reproducible experiment.
- **DVC cache** stores the actual large files by content hash.
